# Decisions

## D-001 技术栈
- 采用微信原生小程序 + TypeScript。
- 使用原生 `Page / Component / App` 组织结构。
- 使用 `.wxml / .wxss(or less) / .ts / .json` 文件体系。

原因：
- 这是 `AGENTS.md` 的硬性约束。
- 当前项目本身也是基于微信原生 TypeScript 模板。

## D-002 Stitch 使用方式
- Stitch 只作为 UI 和布局参考。
- 不直接复用 Stitch 导出的 HTML 作为生产代码。
- Stitch screen 只用于：
  - 页面映射
  - 组件拆分参考
  - 配色、留白、层级参考

原因：
- `AGENTS.md` 明确禁止直接复制 Stitch HTML。
- 当前项目目标是微信原生小程序，不是 Web 页面还原。

## D-003 计分逻辑集中在 utils
- 所有核心计分逻辑集中放在工具层。
- 页面层只负责展示和交互。
- 当前已落地的工具模块包括：
  - `utils/storage.ts`
  - `utils/score.ts`
  - `utils/round.ts`
  - `utils/app-state.ts`

原因：
- `AGENTS.md` 明确要求页面层不要散落核心业务计算。
- 麻将规则复杂，必须集中管理、统一重算、支持撤销。

## D-004 查大叫规则简化
- V1 按 `AGENTS.md` 口径实现：
  - 未听牌者赔付听牌者 `1 番底分`
- 不实现“最大可能番数赔付”。

原因：
- `plain.md` 里对查大叫有“最大可能番数赔付”描述。
- `AGENTS.md` 已把 V1 范围明确收敛为 `1 番底分`，当前以工程约束为准。

## D-005 血流成河范围简化
- V1 只实现：
  - 多次胡牌
  - 一炮多响
  - 手动结束本局
- 暂不实现：
  - 牌墙自动计数
  - 复杂计时逻辑

原因：
- 这是 `AGENTS.md` 对 V1 范围的明确收敛。
- 先完成可走通的主流程，避免在 Phase 2/3 过度扩展。

## D-006 页面架构
- 首页保留为独立 page。
- 当前确认的主页面结构为：
  - `/pages/home`
  - `/pages/round-entry`
  - `/pages/round-confirm`

原因：
- 首页承担规则配置和进入单局的入口职责。
- 结果录入页和确认页与 PRD 用户旅程一致。

## D-007 胡牌登记形态
- 胡牌登记采用组件 modal 形式，不单独做 page。
- 组件路径规划为：`/components/hu-form`

原因：
- 胡牌登记是录入页内的局部流程。
- 小程序里使用组件弹层更符合快速录入体验。

## D-008 玩家名临时修改策略
- 记分板上的玩家名修改只在当前会话内生效。
- 不写回持久化 `playerNames`。
- 展示名与分数字典 key 分离：
  - 展示层使用会话态显示名
  - 分数计算和 storage 继续使用持久化玩家名

原因：
- `plain.md` 明确说明点击玩家名可临时修改，且仅当前会话有效。
- 如果直接修改持久化 `playerNames`，会破坏分数映射和存储语义。

## D-009 最近完成局回看
- 保留 `lastCompletedRound` 作为最近完成局只读回看数据。
- 在首页展示最近完成局摘要卡片。
- 允许用户手动清除该卡片。

原因：
- 确认结算后直接返回首页时，需要保留最近一局的可见反馈。
- 该数据不影响核心计分，只作为结果回看和确认辅助。
